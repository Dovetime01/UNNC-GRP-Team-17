## Context

Virtual production shoots are defined by change. Even during a test shoot — a session intended to validate technical setup rather than capture final footage — stakeholders may ask for things that do not yet exist in the digital environment.

During one such test shoot, the team received a request: a real IV stand present on set needed to appear in the virtual environment for the upcoming shot. The director wanted it modeled as accurately as possible. There was no pre-built asset, no reference sheet, and a limited window before the set would be struck.

The total time available: approximately 20 minutes.

---

## The Workflow

### Step 1 — Photograph the Object

A team member photographed the IV stand from a clear angles using a phone. The photograph below become the input for the AI generation step.

![Original photograph of the IV stand taken on set](/on-set-model-creation/iv-stand-photo.jpg)

No lighting setup was used — the existing set lighting was sufficient for a recognizable photograph. A clean, well-lit image with good separation between the object and its background produces better model quality.

### Step 2 — Upload to Tripo

The photographs were uploaded to [Tripo](https://www.tripo3d.ai/). Tripo accepts a single image and uses its AI pipeline to infer 3D geometry from the 2D input. Processing typically completes within a few minutes.

### Step 3 — Refine and Export

The generated mesh was reviewed in Tripo's viewer. Minor artifacts — extra geometry, incomplete surfaces — were flagged for cleanup. The mesh was exported in a format compatible with Unreal Engine import (`.fbx` is the most common).

![3D model generated by Tripo from the photograph](/on-set-model-creation/tripo-output.jpg)

### Step 4 — Import into Unreal Engine

The mesh was imported into the active UE scene. Standard import checks applied:

- **Scale:** The imported mesh was compared against scene references (e.g., a standard UE mannequin) to verify real-world proportions. Tripo outputs occasionally require uniform scaling.
- **Materials:** A basic material was applied. If the IV stand needed to read well on camera, a closer PBR material was assigned — or at minimum, a material with the correct metallic/roughness response for a stainless steel surface.
- **Positioning:** The stand was placed in the scene and evaluated against the physical set reference visible through the camera feed.

The result was a functional, scene-ready prop.

![IV stand model integrated into the UE scene](/on-set-model-creation/ue-result.jpg)

## What Made This Work

### Pre-Built Process Knowledge

The 20-minute window was only possible because the team already knew which tool to use, how to photograph the input, and what steps to take after export. The decision to reach for Tripo — rather than attempting manual modeling or searching a marketplace — had already been made in advance during setup prep.

### Small Prop, Appropriate Fidelity Bar

The IV stand was a background prop. It did not appear in close-up. This meant the standard Tripo output — a rough but recognizable mesh with basic materials — was sufficient without a full modeling pass.

### On-Hand Technical Capacity

Someone on set knew how to perform the UE import, apply materials, and verify scale. Asset generation and asset integration are two separate skills; having both available in the same session compressed the timeline.

---

## Limitations Observed

Tripo is not a production asset pipeline. Several constraints were visible even in this low-stakes use case:

- **Surface quality:** Fine details — welds, labels, surface wear — are not reliably reconstructed from a single photograph.
- **Scale:** Tripo outputs do not guarantee real-world scale. A manual check is always required.
- **Material inference:** The automatic material assignment is approximate. Production use requires replacing it with a PBR material tuned to the actual surface (in this case, brushed stainless steel).

For a prop requiring close-up readability, this workflow would need additional refinement passes.

---

## Takeaway

AI-assisted asset generation does not replace the modeling department. It creates a third path — between "build it properly" and "go without" — that is fast enough to work in an on-set context when the fidelity bar is appropriate.

The key habit: know which tools exist, know what fidelity they produce, and know when the shot's requirements match what the tool can deliver.
